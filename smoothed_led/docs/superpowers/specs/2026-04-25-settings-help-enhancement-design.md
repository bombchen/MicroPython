# 设置 / 帮助页内容增强设计

- 日期：2026-04-25
- 目标模块：`mobile_app/led_controller/lib/features/settings/presentation/settings_page.dart`
- 相关页面：设备列表页、配网向导页

## 1. 背景

当前设置 / 帮助页只有三段静态说明：

- 配网时需要连接设备热点 `LED_Config`
- 看到“无互联网”属于正常现象
- Android 可能自动切回原 WiFi

这能覆盖最基础提醒，但还不足以支撑真实使用场景。用户在首次配网和失败排查时，仍然缺少一页可以快速查找的操作说明。现有产品也已经具备以下能力：

- 设备列表页提供设置入口
- 配网向导已分步骤引导用户连接设备热点、返回 APP、输入家庭 WiFi、等待设备回到列表
- 配网成功后会返回列表并刷新设备数据

因此，本次增强不是新增复杂功能，而是把设置 / 帮助页补成一个真正可用的单页帮助中心。

## 2. 目标

本次增强需要同时满足以下目标：

1. 帮助首次使用者快速理解完整配网路径
2. 在单页内覆盖高频故障排查
3. 用普通用户能执行的语言解释 Android 和局域网相关限制
4. 保持信息可扫读，不变成长篇说明书
5. 仅保留少量明确动作入口，不把帮助页变成第二个配网向导

## 3. 非目标

以下内容明确不在本次范围内：

- 设备固件版本信息展示
- 反馈入口、客服入口、外链文档
- 多级 FAQ 导航结构
- 插图教程、图片引导、视频引导
- 在帮助页中直接承载完整配网流程操作
- 与现有固件能力无关的通用网络科普

## 4. 页面定位

页面定位为“单页帮助中心”，而不是“设置面板”或“故障工单页”。

设计原则：

- 顶部先服务首次配网
- 中段承接常见问题
- 底部再补系统与权限说明
- 整体偏操作手册风格，优先告诉用户“现在该做什么”

## 5. 信息架构

页面采用单页纵向结构，按以下顺序组织：

### 5.1 顶部摘要区

内容：

- 页面标题
- 一句用途说明
- 一个主按钮：`去添加设备`

要求：

- 摘要区不写大段引言
- 不堆叠多个按钮
- 让用户一眼知道这是“配网帮助入口”

### 5.2 配网步骤

位置：第一页主体，默认展开

建议条目：

1. 连接设备热点 `LED_Config`
2. 连接后返回 APP
3. 输入家庭 WiFi 名称和密码
4. 等待设备重启并回到设备列表

要求：

- 每条只写动作，不写原理
- 每条长度控制在一到两句
- 语气直接，避免解释性修辞

### 5.3 常见故障

位置：第二组，默认收起

建议问题：

1. 看不到 `LED_Config`
2. 手机提示该网络无互联网
3. Android 自动切回原 WiFi
4. 配网完成后列表里没有新设备
5. 进入控制页后设备无响应

要求：

- 问题标题短、可扫读
- 每个问题展开后给出 2 到 4 条可执行操作
- 回答格式以“如果……请……”为主

### 5.4 系统与权限说明

位置：最后一组，默认收起

建议内容：

1. 配网阶段必须手动切换到设备热点
2. Android 的 WiFi 切换不完全由 APP 控制
3. 设备控制要求手机和设备在同一局域网

要求：

- 只解释与当前流程强相关的系统行为
- 解释结果，不展开底层机制
- 文案需能被普通用户直接理解

## 6. 交互设计

### 6.1 折叠方式

采用现成折叠组件实现三组内容：

- `配网步骤` 默认展开
- `常见故障` 默认收起
- `系统与权限说明` 默认收起

不引入自定义复杂动画，不做多级折叠。

### 6.2 动作按钮

只保留一个主动作按钮：

- 位置：顶部摘要区
- 文案：`去添加设备`
- 行为：跳回现有添加设备 / 配网页入口

本页不额外增加：

- FAQ 内嵌按钮组
- “重新开始配网”等第二主按钮
- 打开系统设置的快捷动作

原因：帮助页应当是说明入口，而不是流程控制页。

## 7. 文案规范

整体文案采用“操作手册型”风格。

要求如下：

- 标题尽量用动词开头
- 句子尽量短
- 优先写动作结论，不先讲背景
- 避免“可能、大概、建议你可以尝试”这类弱语气

示例方向：

- 写法：`连接设备热点 LED_Config`
- 不写：`为了继续后续配网流程，你需要先尝试连接一个名为 LED_Config 的热点`

- 写法：`看到“无互联网”提示属于正常现象，可继续连接`
- 不写：`因为设备热点本身不提供外网访问能力，所以 Android 可能认为它不可用`

## 8. 内容边界

为了保持页面可扫读，条目数量需要被限制：

- 配网步骤：4 条左右
- 常见故障：最多 5 条
- 系统与权限说明：最多 3 条

如果后续还要补内容，优先方式应当是替换低价值条目，而不是持续加长页面。

## 9. 对现有代码的影响

本次增强应尽量复用现有页面结构，不引入新的复杂状态管理。

建议实现边界：

- 保持 `SettingsPage` 为轻量展示页
- 主要改动集中在内容结构和组件分组
- 仅增加必要的导航回调
- 不新增额外业务 controller

## 10. 测试要求

Widget test 应覆盖以下关键点：

1. 页面能看到顶部摘要与主按钮
2. `配网步骤` 分组默认展开
3. `常见故障` 分组存在，且至少一个问题标题可见
4. `系统与权限说明` 分组存在
5. 主按钮能触发预期导航行为

测试策略：

- 只断言关键标题、关键按钮和少量关键短句
- 不对整段长文案做全文匹配
- 允许后续微调文案时不大量改测试

## 11. 风险与控制

### 风险 1：页面重新变成长文说明

控制方式：

- 限制每组条目数量
- 默认只展开第一组
- 将解释性内容压缩到系统说明分组

### 风险 2：帮助页与配网向导职责重叠

控制方式：

- 帮助页只解释流程，不接管流程
- 只保留一个“去添加设备”入口
- 不在帮助页里复制完整向导交互

### 风险 3：文案变得太技术化

控制方式：

- 只保留用户能执行的判断和动作
- 删除底层协议、广播、端口等术语
- 所有说明以“用户现在该做什么”为检查标准

## 12. 验收标准

满足以下条件即可认为本次增强完成：

1. 设置 / 帮助页从三段静态文本升级为结构化帮助中心
2. 顶部先展示配网相关摘要和唯一主动作
3. 页面完整覆盖配网步骤、常见故障、系统说明三组内容
4. 内容风格保持短句、直接、可执行
5. 用户从帮助页返回后，能够无歧义地继续现有配网流程
