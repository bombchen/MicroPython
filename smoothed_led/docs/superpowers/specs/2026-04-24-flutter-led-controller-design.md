# Flutter LED 控制器设计文档

日期：2026-04-24

## 1. 目标

开发一个基于 Flutter 的 Android APP，用于控制运行 `smoothed_led` MicroPython 固件的 ESP8266 设备。

该 APP 需要覆盖两类核心任务：

1. 通过固件现有的 AP + UDP 配置流程完成设备首次配网。
2. 在局域网内通过固件现有的 UDP 控制协议完成日常控制。

这是一份面向 `v1` 的可发布首版设计文档，不是演示原型，也不是追求企业级完整度的近生产级方案。

## 2. 固定范围

### 2.1 范围内

- Android 优先发布。
- 使用 Flutter 开发。
- 半自动配网流程。
- 支持多设备，但仅平铺列表，不做分组。
- 配网成功后自动登记设备。
- 仅支持局域网内控制。
- 单设备控制页采用中等强度状态反馈。
- 支持设备重命名与本地记录删除。
- 提供配网帮助、网络异常排查与权限说明。

### 2.2 范围外

- iPhone 优先，或为 iOS 单独保证首版配网体验。
- 通用局域网自动发现。
- 手动输入 IP 添加设备。
- 房间、分组、场景。
- 互联网远程控制。
- 收藏、预设、批量控制。
- 颜色选择器或自定义颜色编辑。
- 在本次 APP `v1` 中修改固件行为。

## 3. 固件约束

设计必须严格匹配当前 `README.md` 中记录的固件行为。

### 3.1 配置模式

- 设备会开启 AP：`LED_Config`
- APP 需要通过 UDP `8889` 端口完成配置
- 当前支持的配置命令：
  - `config:SSID:PASSWORD`
  - `status`
  - `list`

### 3.2 控制模式

- 设备在 UDP `8888` 端口监听控制命令
- 当前支持的控制命令：
  - `mode:<effect>`
  - `mode:next`
  - `mode:prev`
  - `bright:<0-255>`
  - `status`
  - `help`

### 3.3 支持的灯效

- `rainbow`
- `breath`
- `fire`
- `starry`
- `wave`
- `chase`
- `sparkle`
- `snake`

## 4. 产品方向

APP 采用“设备列表优先”的信息架构。

原因如下：

- 它最贴合“多设备但不分组”的已确认范围。
- 能把首次配网流程收束在一个独立向导里。
- 如果后续固件能力扩展，结构也能平滑演进。
- 不会在首页层面承诺当前固件并不支持的控制能力。

## 5. 信息架构

APP 由四个顶层区域组成：

1. 设备列表
2. 添加设备向导
3. 设备控制页
4. 设置与帮助

### 5.1 设备列表

这是默认首页。

职责：

- 展示所有已登记设备
- 展示最近一次已知在线状态
- 展示最近一次已知 IP
- 展示最近同步时间
- 提供“添加设备”入口
- 提供进入单设备控制页的入口

行为：

- 如果当前没有设备，则展示明确的空状态页和配网入口
- 如果已有设备，则以平铺列表展示
- `v1` 不引入房间、分组或总控台抽象

### 5.2 添加设备向导

这是首配专用流程。

职责：

- 告知用户当前即将执行的操作
- 引导用户连接 `LED_Config`
- 收集目标家庭 WiFi 的信息
- 通过 UDP 发送配置
- 等待设备重启并重新入网
- 成功后将设备登记到本地列表

### 5.3 设备控制页

这是单设备控制页面。

职责：

- 展示设备身份与连接状态
- 页面进入时拉取当前设备状态
- 支持切换灯效
- 支持调整亮度
- 支持 `next` 和 `prev`
- 支持手动刷新状态
- 支持重命名与删除本地设备记录

### 5.4 设置与帮助

职责：

- 解释权限与 Android 网络行为
- 提供配网失败排查
- 提供网络异常帮助
- 展示 APP 信息与版本信息

## 6. 关键用户流程

### 6.1 首次配网流程

1. 用户打开 APP。
2. 用户在设备列表页看到空状态。
3. 用户点击 `添加设备`。
4. APP 进入引导式配网流程。
5. APP 解释设备会暴露 `LED_Config` 热点。
6. APP 打开 Android WiFi 设置页或引导用户切换到系统网络页。
7. 用户连接 `LED_Config`。
8. 用户返回 APP 并确认继续。
9. APP 收集家庭 WiFi 的 SSID 和密码。
10. APP 向 UDP `8889` 发送 `config:SSID:PASSWORD`。
11. APP 等待设备重启并重新进入局域网。
12. APP 在当前配网会话内执行一次受限的局域网解析，用来识别刚刚完成配网的设备 IP。
13. 成功后 APP 将设备登记到本地，并返回设备列表页。

### 6.2 日常控制流程

1. 用户打开设备列表页。
2. 用户点击某个已登记设备。
3. APP 进入设备控制页。
4. APP 请求 `status`。
5. APP 展示当前模式、亮度和在线状态。
6. 用户发送模式切换或亮度调整命令。
7. APP 在操作后刷新一次状态。
8. APP 明确反馈成功、超时或离线状态。

## 7. 配网体验设计

由于 Android WiFi 切换是 `v1` 中最容易出问题的环节，因此配网流程必须比最小向导更显性、更可恢复。

### 7.1 向导形态

采用明确的步骤式任务流，并展示进度状态：

1. 准备设备
2. 连接 `LED_Config`
3. 返回 APP
4. 输入家庭 WiFi
5. 等待重启并完成

### 7.2 交互原则

- 不假装 WiFi 切换可以完全自动完成
- 始终明确展示当前步骤和下一步动作
- 提供系统 WiFi 跳转按钮和“我已连接，继续”这类显式操作
- 出错后停留在当前流程内，不直接把用户踢回首页
- 在允许的前提下保留已输入 SSID，便于重试

### 7.3 必须出现的引导文案

向导必须明确告诉用户：

- `LED_Config` 可能会显示为“无互联网”的网络，这是正常现象
- Android 可能会自动切回其他已保存 WiFi
- 如果没有看到 `LED_Config`，设备可能不在配置模式，或已经连入其他网络
- 连接设备热点后再返回 APP 是预期行为，不是异常流程

### 7.4 失败恢复

向导必须覆盖以下失败场景，并给出明确恢复动作：

- 连接 `LED_Config` 失败
  - 动作：停留在当前步骤，允许再次打开 WiFi 设置
- 发送配置命令失败
  - 动作：保留当前表单内容，允许原地重试
- 设备重启后未重新回到局域网
  - 动作：允许继续等待一次，或重新开始配网

### 7.5 配网会话内的设备 IP 解析

`v1` 不支持“通用局域网自动发现”作为产品功能，但配网成功后仍然需要一个受限步骤来获得设备 IP。

规则如下：

- APP 在发送 WiFi 配置并等待设备重启后，可以在当前配网会话内发起一次短时 UDP 广播探测，目标端口为 `8888`
- 该探测只允许在一个进行中的配网会话内使用
- 在配网时间窗口内，第一个返回有效响应的设备被视为刚刚完成配网的目标设备
- 响应源 IP 将被保存为该设备的 `ipAddress`

之所以不把它定义为“通用自动发现”，原因是：

- 它只会在用户主动执行配网之后触发
- 它只在一个很短的配网时间窗口内存在
- 它不会在正常使用时扫描或维护任意局域网设备列表

## 8. 设备控制页设计

控制页应该务实，且严格贴合当前固件能力。

### 8.1 必备元素

- 设备名称
- 最近一次已知 IP
- 在线 / 离线 / 超时状态
- 最近同步时间
- 当前模式展示
- 亮度滑杆
- 所有已支持灯效的选择区
- `上一个灯效`
- `下一个灯效`
- `刷新状态`
- 重命名设备
- 删除设备记录

### 8.2 明确不做

以下内容不进入 `v1` 控制页：

- 颜色编辑器
- 场景编辑器
- 组控制
- 收藏预设
- 任何当前固件并不支持的控制抽象

## 9. 状态反馈模型

APP 使用“中反馈”模型，而不是伪实时同步。

### 9.1 设备列表

- 页面进入时刷新最近一次已知状态
- 配网页返回后刷新
- 用户主动下拉或点击刷新时更新
- 不做高频后台轮询

### 9.2 设备控制页

- 页面进入时请求一次 `status`
- 每次用户控制操作后刷新一次状态
- 明确区分 `online`、`offline`、`sending`、`timeout`

之所以采用这种方式，是因为当前固件基于 UDP，且没有可靠的服务端推送状态机制。

原因：

- 比较适合 Flutter 分层结构
- 能把设备列表、配网向导和控制页状态清晰拆开
- 能较好处理异步网络请求与本地持久化

### 12.1 必须拆分的状态域

- 设备列表状态

### 12.2 配网状态机

## 13. 网络设计

### 13.1 配网传输

- 使用 UDP `8889` 端口进行配置
- 严格按固件已有命令格式发送配置
- 在需要时可使用 `status` 或 `list` 辅助配网过程

### 13.2 控制传输

- 使用 UDP `8888` 端口进行控制
- 支持 `status`、`mode:*`、`mode:next`、`mode:prev`、`bright:*`

### 13.3 配网后设备解析传输

- 配网配置发送完成后，APP 可以在短时间内向 UDP `8888` 广播 `status` 探测
- 活跃配网窗口中收到的有效响应用于识别该设备 IP
- 如果在限定窗口内没有收到有效响应，则视为配网尚未完成，向用户展示恢复动作

### 13.4 可靠性预期

UDP 天生不可靠。

因此：

- 每个关键操作都必须有超时处理
- UI 不能表达“命令一定送达”
- 重试路径必须对用户可见
- 如果有响应证据，应以响应结果作为成功依据

## 14. 持久化设计

本地持久化需要保存已登记设备列表。

### 必须保存的字段

- 设备身份信息
- 用户定义的设备名称
- 最近一次已知 IP
- 最近一次状态快照
- 最近同步时间

### 持久化职责

- APP 重启后仍然保留已登记设备
- 支持重命名与删除
- 成功拉取状态后，更新最近一次设备状态

## 15. 错误处理

错误提示必须是“可执行的”，而不只是报错文本。

### 15.1 配网错误

- 找不到 `LED_Config`
- 用户没有成功切换网络
- 配置发送超时
- 设备重启等待超时
- 设备没有重新进入局域网

每种错误都要映射到明确的下一步动作，例如：

- 重试当前步骤
- 再次打开 WiFi 设置
- 再等待一次
- 重新开始配网

### 15.2 控制错误

- 设备离线
- 设备 IP 失效或过期
- UDP 命令超时
- 状态读取失败

每种错误都要映射到明确的下一步动作，例如：

- 刷新状态
- 稍后重试
- 必要时删除设备并重新配网

## 16. 权限与平台注意事项

由于 `v1` 以 Android 为优先平台，因此必须显式处理：

- WiFi / 网络行为说明
- 配网期间 APP 前后台切换
- 系统设置页跳转行为
- 所需权限以及这些权限为什么需要

设置与帮助页必须用普通用户能理解的语言解释这些内容。

## 18. 成功标准

当满足以下条件时，可认为 `v1` 达到目标：

- 用户无需外部工具即可完成新设备配网
- 成功配网的设备能够出现在设备列表中
- 用户能在同一局域网内打开控制页并可靠发送已支持命令
- 配网或控制失败时，APP 能给出可理解、可执行的反馈
- 整体产品完成度足以作为 Android 首版发布
