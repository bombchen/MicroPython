# Flutter LED 控制器产品化 UI 改造设计

日期：2026-05-17  
项目：`smoothed_led` / `mobile_app/led_controller`  
目标：将当前 Flutter 控制端从“可用的工程界面”改造成更接近正规家居消费品的产品界面

## 目标

在不改变现有核心业务能力的前提下，重做 Flutter 控制端的视觉系统、页面层级和交互呈现方式，让 APP 更像一个面向普通家庭用户的正式产品，而不是开发演示或设备调试工具。

本次改造的核心目标包括：

- 建立统一、明确的品牌化视觉语言
- 让首页、控制页、配网页和帮助页都具备清晰的产品层级
- 降低当前界面的工程感、默认 Material 感和技术说明感
- 保留现有配网、控制、重命名、删除、状态刷新等能力

## 已确认方向

- 产品气质选择为“家居消费品”
- 改造范围选择为“视觉、结构、文案、状态反馈的整套体验”
- 推荐实施策略为“消费品化重构”，即在保留现有页面职责和主要流程的前提下，重做视觉层和信息结构

## 已确认约束

- 不修改固件协议和设备控制能力边界
- 不新增远程控制、房间、场景、自动化等新产品能力
- 保留当前 Flutter 路由和状态管理方式
- 保留当前配网状态机和控制器的大体逻辑
- Android 仍然是首要目标平台
- 优先通过表现层重构提升产品感，而不是做业务侧大重写

## 非目标

- 不重做应用的领域模型或数据层
- 不引入新的状态管理框架
- 不增加登录、云同步、场景收藏、房间分组等新能力
- 不修改 UDP 协议格式
- 不把 APP 产品方向改成“灯光氛围社区”或“内容型应用”
- 不追求品牌营销级的复杂动画或高成本插画体系

## 现状问题

### 全局层面

- 全局主题几乎是默认 `Material 3`，缺少明确的品牌感
- 视觉语言没有形成统一规则，页面之间像多个可用界面的拼接
- 背景、卡片、按钮、输入框、状态提示都缺少统一层级

### 信息结构层面

- 首页更像设备数据库列表，而不是“家庭灯光首页”
- 控制页更像功能面板，缺少主操作优先级
- 配网页更像网络配置说明书，而不是面向普通用户的引导流程
- 帮助页更像 FAQ 折叠列表，而不是支持中心

### 文案与反馈层面

- 当前文案偏工具型和技术说明型
- 异常、等待、成功状态的反馈表达不够产品化
- 页面缺少对用户的情绪照顾和使用信心建立

## 推荐方案

推荐采用“表现层产品化重构”方案。

该方案保留现有功能边界和业务控制器，只重做：

- 全局主题
- 页面骨架
- 组件表现
- 状态反馈样式
- 文案语气

这是在投入成本、回归风险和产品提升效果之间最平衡的方案。

## 备选方案

### 1. 轻量换肤

做法：

- 只调整颜色、圆角、按钮和字体样式
- 保持页面结构几乎不变

优点：

- 改动最小
- 回归风险最低

缺点：

- 只能让界面“没那么像 demo”
- 难以解决页面结构本身的产品感缺失

### 2. 表现层产品化重构

做法：

- 保留现有页面职责和主流程
- 重做全局主题、页面层级、主次操作关系和反馈样式

优点：

- 最适合当前产品阶段
- 能明显提升正规产品感
- 不需要大规模改动业务逻辑

缺点：

- 不是简单换皮，需要重写多页布局

### 3. 场景化品牌体验重设计

做法：

- 把应用核心从“设备控制”改成“氛围体验”
- 首页强调情绪、预设、房间和场景

优点：

- 品牌感最强

缺点：

- 与当前功能能力不匹配
- 重设计范围过大

结论：

本次采用方案 2，即“表现层产品化重构”。

## 高层设计

### 产品定位

应用定位为“家中的灯光设备控制器”，不是开发工具，也不是重内容消费的氛围社区。

界面气质应体现：

- 温和
- 整洁
- 可信
- 易上手
- 有轻微的生活方式感，但不过度情绪化

### 页面角色

- 设备列表页：家庭灯光首页
- 设备控制页：单设备遥控面板
- 配网页：陪伴式设备添加流程
- 帮助页：支持中心

### 实现边界

本次只在表现层做系统性重构：

- 允许拆分私有 UI 组件
- 允许重写现有页面布局
- 不要求重构已有控制器逻辑

## 视觉系统设计

### 色彩方向

采用偏暖、柔和、家居感的色彩系统。

建议方向：

- 主色：暖杏橙 / 柔和琥珀
- 辅色：浅米白、暖灰、低饱和雾蓝
- 成功色：柔和绿色
- 警告色：暖黄色
- 错误色：低饱和砖红或柔红
- 页面背景：浅暖灰，而不是纯白满屏

### 组件语言

- 卡片作为页面主要信息容器
- 大圆角，建立柔和的消费品界面感
- 主按钮饱满、明确，作为页面主操作锚点
- 输入框、状态卡、空状态卡、帮助卡遵循统一圆角和留白规则
- `ChoiceChip` 不再承担主视觉角色，只作为次级选择器

### 排版层级

建立统一的三层排版：

- 页面级标题
- 区块标题
- 辅助说明

避免当前“所有文字都像说明文本”的扁平化问题。

### 状态表达

空状态、等待状态、成功状态、失败状态都需要具备更完整的呈现：

- 状态标题
- 简短解释
- 推荐下一步
- 简单图形元素或更明确的状态容器

## 页面设计

### 设备列表页

目标：

从“设备列表”改造成“家庭灯光首页”。

设计要点：

- 顶部显示更像首页欢迎区的头部，而不是仅有普通标题栏
- 页面上方提供主要 CTA，例如“添加设备”
- 设置/帮助入口弱化为辅助入口
- 空状态使用完整卡片，而不是简单文字居中
- 有设备时采用卡片式设备项，而不是 `ListTile + Divider`
- 每张设备卡显示最重要信息：
  - 设备名
  - 在线状态
  - 当前模式或亮度摘要
  - 最近同步时间

非目标：

- 不引入房间、分组或批量控制

### 设备控制页

目标：

从“参数堆叠页”改造成“主操作优先的设备遥控页”。

设计要点：

- 顶部建立设备头卡，显示名称、状态、最近同步等摘要
- 第一屏优先展示当前模式和亮度控制
- 常用灯效以更强视觉表达展示，如推荐模式卡片
- 次要灯效和辅助控制往下收
- `上一个灯效` / `下一个灯效` 降为次级操作
- 管理型动作如重命名、删除进入更多菜单
- 离线、超时、发送中、异常等状态通过状态卡展示

保留能力：

- 刷新状态
- 切换灯效
- 调整亮度
- 重命名
- 删除本地记录

### 配网页

目标：

从“网络配置说明书”改造成“陪伴式添加设备向导”。

设计要点：

- 持续显示当前步骤进度
- 每一步聚焦一个主要动作
- 文案拆成主句和辅助句，降低技术说明感
- 等待页面增加陪伴式反馈，而不只是转圈和秒数
- 失败页提供更明确的恢复动作
- 成功页提供更完整的完成感和确认出口

必须保留的现实约束表达：

- `LED_Config` 无互联网是正常现象
- Android 可能自动切回其他 WiFi
- 某些步骤需要用户手动进入系统 WiFi 设置

### 帮助页

目标：

从“FAQ 折叠页”改造成“支持中心”。

设计要点：

- 顶部先给出支持中心的用途说明
- 以两个主要动作作为第一视觉层：
  - 添加设备
  - 重置设备配网
- 常见问题、配网步骤和系统说明收纳在下方
- 降低 `ExpansionTile` 单一结构的页面占比

## 文案策略

### 语气目标

从“功能说明型”切换到“产品陪伴型”。

要求：

- 更短
- 更直白
- 更让普通用户安心

### 文案原则

- 主文案告诉用户当前该做什么
- 辅助文案解释可能出现的正常现象
- 失败文案先说明问题，再给推荐动作
- 避免工程师口吻和过多底层原因解释

示例方向：

- “设备响应异常，请稍后重试”  
  调整为更产品化的表达，如“设备暂时没有响应，你可以稍后重试”

## 实现设计

### 允许改动的文件

- `mobile_app/led_controller/lib/app/app.dart`
- `mobile_app/led_controller/lib/features/devices/presentation/device_list_page.dart`
- `mobile_app/led_controller/lib/features/devices/presentation/device_control_page.dart`
- `mobile_app/led_controller/lib/features/pairing/presentation/pairing_page.dart`
- `mobile_app/led_controller/lib/features/settings/presentation/settings_page.dart`

必要时允许：

- 在相关页面文件中提取私有小组件
- 新增有限的表现层辅助组件文件

### 全局主题

`app.dart` 需要从简单的 `ThemeData(colorSchemeSeed: Colors.orange)` 升级成更明确的主题系统，包括：

- `ColorScheme`
- 卡片主题
- 按钮主题
- 输入框主题
- `AppBar` 主题
- 页面背景色
- 文字层级

### 设备列表页

保留当前数据流和刷新逻辑，只重写表现层结构。

### 控制页

保留控制器和状态同步逻辑，重点重写布局结构和操作分层。

### 配网页

保留当前配网状态机，重点重写各步骤的可视呈现和文案。

### 帮助页

保留现有功能动作，重写页面层级和信息组织方式。

## 推荐实施顺序

1. 全局主题
2. 设备列表页
3. 设备控制页
4. 配网页
5. 帮助页

原因：

- 先统一视觉系统，后续页面改造才不会各自为战
- 设备列表页是第一印象，收益最高
- 控制页是高频核心页面，优先级次高
- 配网页和帮助页重要，但使用频率低于首页和控制页

## 测试与验证设计

### 目标

确保 UI 改造后：

- 现有功能仍可用
- 现有测试大体可维护
- 页面层级明显改善

### 验证范围

- 现有 widget tests 根据新 UI 调整
- 现有配网、控制、帮助流程测试继续通过
- 页面在常见手机宽度下没有布局溢出

### 手工验证重点

1. 空状态首页是否像正式产品首页
2. 有设备时首页是否能快速识别设备状态
3. 控制页第一屏是否能快速完成常用操作
4. 配网页是否更容易理解当前步骤
5. 帮助页是否更像支持中心而不是设置列表

## 风险与取舍

### 风险

- 页面重构会导致现有 widget tests 大量更新
- 单文件页面如果继续堆组件，维护性会下降
- 如果视觉系统定得不够清晰，页面容易再次出现“局部变美、整体仍散”的问题

### 取舍

- 不做能力扩张，只提升表现层完成度
- 不追求品牌营销级大动画，优先做稳定、清晰、可信的体验
- 在保持当前业务逻辑稳定的前提下接受中等强度的页面结构调整

## 成功标准

- APP 首页、控制页、配网页、帮助页都具备统一的产品化视觉语言
- 用户第一眼感知从“工程工具”转变为“家居消费品控制器”
- 最常用操作在页面上有明确主次层级
- 现有核心功能不回归
- 测试与手工体验都能支持“这是可上架的正规产品界面”这一判断
